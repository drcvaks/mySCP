mySCP Project Summary
Last updated: June 30, 2026

Purpose
-------
mySCP is a cross-platform Expo app for a Semichas Chaver Program-style learning community. The goal is to help participants join a local chaburah, access source sheets and review material, submit questions to their Rav, and review for bechinos. It also needs practical tools for rabbis, local admins, and global admins to manage the program without touching Supabase manually.

The app should feel polished, calm, professional, and easy to scan. The visual direction is based on the cleaner ReplitSCP reference, but mySCPcodex remains the source of truth. The current palette is restrained navy, gold, white, and light gray, with compact cards, clear hierarchy, readable typography, and mobile-friendly tap targets.

Current Tech Stack
------------------
- Expo SDK 54
- Expo Router
- TypeScript
- React Native / React Native Web
- Supabase Auth
- Supabase Postgres
- Supabase Row Level Security
- Supabase RPC functions for sensitive workflows
- Supabase Storage bucket planned/partly configured for learning files

Important dependency notes:
- Keep Expo SDK 54.
- Do not introduce package changes unless necessary.
- `@supabase/supabase-js` is installed.
- `expo-document-picker` is installed for native file selection/upload.
- `expo-file-system` is installed so native uploads can read selected files before sending them to Supabase Storage.
- `react-native-url-polyfill` is installed and imported in the Supabase client for Expo/React Native compatibility.
- `.env` is local and ignored.
- `.env.example` is tracked with placeholders only.
- Never put service-role keys, database passwords, or direct Postgres connection strings in the Expo app.

Environment Variables
---------------------
The Expo client reads:

EXPO_PUBLIC_SUPABASE_URL=https://your-project-ref.supabase.co
EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY=your-publishable-key

Use the Supabase publishable/anon key only. The service role key must never be exposed in the app.

Product Roles
-------------
Participant:
- signs in/signs up with first name, last name, optional city, email, and password
- joins or requests to join one chaburah
- views Dashboard, My Chaburah, Files, Review, Directory, Ask Rav, Profile, Settings
- submits Ask the Rav questions
- reviews published questions

Local Rabbi:
- belongs to a specific chaburah through `chaburah_members.member_role = 'rabbi'`
- has `profiles.role = 'local_rabbi'`
- sees Rabbi Hub
- answers Ask the Rav questions for their chaburah
- creates/edits/enables/disables review questions for their chaburah
- sees Rabbi Hub in mobile bottom navigation

Local Admin:
- belongs to a specific chaburah through `chaburah_members.member_role = 'admin'`
- has `profiles.role = 'local_admin'`
- manages local chaburah settings, join requests, members, and URL-based files

Global Admin:
- has `profiles.role = 'global_admin'`
- can create/activate/deactivate chaburos
- can search chaburos in Global Admin
- can promote another global admin or reset a user to participant
- can select which chaburah the Admin screen is managing
- can assign local rabbis/admins to a selected chaburah by email
- can see Rabbi Hub in mobile bottom navigation

Role Design Decision
--------------------
Local rabbi/admin assignment belongs in Admin because it must be tied to a specific chaburah. Global Admin's app-wide role tool should only promote Global Admin or reset to Participant. Local Rabbi/Local Admin should not be assigned from Global Admin's role selector.

Navigation
----------
Web/tablet:
- left-side compact navigation

Android/iOS phone:
- compact bottom navigation
- participants see: Dashboard, My Chaburah, Review, Files
- rabbi/global admin users also see Rabbi Hub at the bottom
- hamburger menu in upper left contains remaining sections such as Directory, Admin, Global Admin, Profile, Settings, etc., depending on role

Screens
-------
Dashboard:
- overview of current chaburah, source sheet/review prompts, Ask Rav action, etc.

My Chaburah:
- selected/current chaburah details
- read-only active member roster with names, roles, and joined dates
- local announcements
- source sheets/recordings
- review assigned count
- Ask Rav entry

Directory:
- searchable active chaburah directory
- shows only chaburos with `status = active`
- shows Join / Request Join / Pending / Joined
- if join requires approval, user sees that approval is required
- changing chaburah means leaving prior participant memberships

Files:
- searchable and filterable
- filters: Scope, Type, Week
- files have Coverage: Week, Bechina Review, or Entire Zman
- opens external URLs or signed Supabase Storage URLs
- Admin publishing supports either external URLs or native upload to the private Supabase Storage bucket

Review:
- week selector is generated from current week plus future range and from actual published question weeks
- current review week is stored in `src/shared/reviewWeeks.ts`
- questions are loaded from Supabase
- answer checking uses RPC so answer keys are not exposed
- review sessions/results are saved to Supabase

Ask Rav:
- participant submits questions for current chaburah
- local questions/history show in the screen

Rabbi Hub:
- answers submitted Ask the Rav questions
- publishes review questions
- edits existing review questions
- enables/disables review questions
- filters managed review questions by selected week
- question form:
  - defaults to current week
  - week chips extend three weeks beyond current week
  - supports True/False and Multiple Choice
  - Multiple Choice asks how many options, then shows one input per option
  - correct-answer buttons remain compact as A/B/C/D instead of showing long answer text
  - topic is hidden from form but saved as `Week X Review`
- answer keys live in protected `review_question_answers`

Admin:
- local chaburah tools
- for Global Admin, top selector chooses which chaburah Admin controls
- local admin/rabbi users manage only their assigned/current chaburah
- structured chaburah settings:
  - address
  - contact email
  - day of week
  - time
  - AM/PM
  - meeting/Zoom link
  - description
  - discussion setting
  - join approval setting
- join request review:
  - approve
  - reject
- member management:
  - search by name, email, role, or status
  - filter by all/active/pending/suspended/left
  - suspend active members
  - reactivate suspended/left participant memberships
  - remove participant memberships by marking them left
- assign local leadership:
  - Global Admin only
  - enter user email
  - choose Rabbi or Local Admin
  - RPC creates/updates membership and profile role
- publish URL-based learning files
- recent files list

Global Admin:
- create chaburah
- structured schedule and contact email on create
- search chaburos before activating/deactivating
- active/inactive badges
- promote global admin
- reset user to participant
- live overview currently only shows configured chaburos count
- confusing review-session analytics count was removed until better analytics exists

Profile:
- users can edit first name, last name, and city
- email, role, and current chaburah are shown read-only

Settings:
- placeholder

Status Messages
---------------
Use the shared `StatusBanner` component for success, info, and error messages. Detailed Supabase/RPC/error text should remain visible during testing so it can be pasted into Codex for debugging. The improvement is presentation, not hiding errors.
Error status messages also trigger a cross-platform alert popup so failures are visible on long screens. Success/info messages remain banner-only.

Security Requirements
---------------------
- RLS must remain enabled.
- Client uses publishable/anon key only.
- Service role key never goes into Expo/web/mobile code.
- Review answer keys must not be exposed in `review_questions`.
- Review answers are checked by RPC.
- `review_question_answers` access is restricted to managers/global admins.
- Local rabbi/admin permissions come from `chaburah_members`, not just text fields.
- `chaburos.rabbi_name` is display text only and does not grant permissions.
- Assigning a rabbi/admin requires membership row plus profile role.
- Global Admin should not impersonate local admins; instead, Admin screen scopes tools to selected chaburah.
- Inactive chaburos should not appear in Directory.

Seed Data Notes
---------------
`supabase/seed.sql` includes test chaburos, announcements, learning files, review questions, and answer keys. It expects at least one global admin profile because seeded content needs an author.

Known Supabase Setup Notes
--------------------------
- If a verification email redirects to `http://localhost:3000`, update Supabase Authentication URL Configuration.
- During development, email confirmation can be disabled or users can be manually confirmed.
- Production should configure proper web URL and/or mobile deep links.
- If Supabase says an RPC cannot be found in schema cache after running SQL, try:
  notify pgrst, 'reload schema';
